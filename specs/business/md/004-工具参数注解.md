# 工具参数注解

> 本文沉淀工具参数的定义、校验与 Schema 生成的注解驱动设计方案。核心结论：不做通用 JSON Schema 对象模型，采用注解声明 + 反射扫描，Schema 生成器默认输出标准 JSON Schema，LLM Client 层做提供商适配。

## 核心结论

**注解驱动的 POJO，不做中间通用 Schema 对象模型。**

工具参数的约束信息通过注解声明在 POJO 类上。需要什么扫什么——Schema 生成器、校验器、Jackson 反序列化各自独立扫描注解，不缓存、不共享中间对象模型。默认产出标准 JSON Schema，不同 LLM 提供商的格式适配由 LLM Client 层负责。

## 注解定义

### @ToolParam

标注在参数类上，声明这是一个工具的参数对象。

```java
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.TYPE)
public @interface ToolParam {
    /** 工具参数的中文描述 */
    String name();
}
```

### @ToolField

标注在参数类的字段上，声明字段名、描述、校验规则。字段的 JSON Schema 类型由 Java 字段类型通过反射自动推导，不需要显式声明。

```java
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.FIELD)
public @interface ToolField {
    /** 字段名 */
    String name();

    /** 字段描述 */
    String desc();

    /** 校验规则列表，按声明顺序执行 */
    Class<? extends Rule>[] rule() default {};

    /** 自定义类型转换器，覆盖默认的类型推导逻辑 */
    Class<? extends TypeConverter> typeConverter() default TypeConverter.class;
}
```

### 类型推导与转换器

不维护独立的 `FieldType` 类型体系。字段的 JSON Schema 类型由 Java 字段类型通过反射直接推导，避免维护两套并行类型系统。

#### TypeConverter

类型转换器接口，负责将 Java 字段类型映射为 JSON Schema 片段（`JSONObject`）。

```java
public interface TypeConverter {
    /**
     * 将 Java 字段类型转换为 JSON Schema 片段
     * @param field 字段反射对象，可从中获取泛型参数信息
     * @param registry 类型转换器注册中心，用于递归处理嵌套类型
     * @return JSON Schema 片段，如 {"type": "string"}
     */
    JSONObject convert(Field field, TypeConverterRegistry registry);
}
```

#### TypeConverterRegistry

类型转换器注册中心，维护 `Java 类型 → TypeConverter` 的映射关系。

```java
public interface TypeConverterRegistry {
    /**
     * 注册一个类型转换器
     * @param javaType  Java 类型
     * @param converter 对应的转换器
     */
    void register(Class<?> javaType, TypeConverter converter);

    /**
     * 查找类型对应的转换器
     * @param javaType Java 类型
     * @return 对应的转换器，未找到返回 null
     */
    TypeConverter lookup(Class<?> javaType);
}
```

#### 转换流程

1. Schema 生成器遍历 `@ToolField` 字段
2. 检查字段上是否指定了自定义 `typeConverter`，有则直接使用
3. 否则通过 `Field.getGenericType()` 获取字段的泛型类型，到注册中心查找匹配的转换器
4. 对于 `List<String>` 等泛型类型，转换器内部递归调用注册中心处理泛型参数
5. 转换器返回 `JSONObject`，Schema 生成器直接拼入最终 JSON Schema

#### 默认支持的 Java 类型映射

| Java 类型 | JSON Schema 输出 |
|-----------|-----------------|
| `String` | `{"type": "string"}` |
| `int`、`long` | `{"type": "integer"}` |
| `double`、`float` | `{"type": "number"}` |
| `boolean` | `{"type": "boolean"}` |
| `List<T>`、`Set<T>` | `{"type": "array", "items": {...}}`（递归推导 T） |
| `Map<String, Object>` | `{"type": "object"}` |
| 自定义枚举 | `{"type": "string", "enum": [...]}` |

特殊格式（如 URL、Email）通过 `@ToolField.typeConverter` 指定自定义转换器实现。

### Rule

校验规则接口，每个实现类负责一项具体的校验逻辑。校验器拿到字段值后，按 `@ToolField.rule()` 的顺序遍历执行，任一失败立即返回错误。

```java
public interface Rule {
    /**
     * 校验字段值
     * @param value 字段原始值（Array 类型时为 List）
     * @return true 表示通过，false 表示失败
     */
    Boolean validate(Object value);
}
```

## 示例：ReadFileParams

```java
@ToolParam(name = "读文件工具的参数")
public class ReadFileParams {
    @ToolField(
        name = "path",
        desc = "文件的路径，支持相对路径和绝对路径",
        rule = { RequiredRule.class }
    )
    private String path;

    @ToolField(
        name = "lines",
        desc = "读取文件的具体行，如 '112' 或 '112-119'",
        rule = { LinesRule.class }
    )
    private List<String> lines;
}
```

### LinesRule 校验逻辑

- `"112"` → 合法（单行）
- `"112-119"` → 合法（行范围，起始 ≤ 结束）
- `"119-112"` → 非法（起始 > 结束）
- `"112~119"` → 非法（分隔符不是 `-`）

## 整体流程

```
LLM 返回 JSON
      │
      ▼
Jackson 反序列化 → POJO 实例
      │
      ▼
Rule 链校验（按 @ToolField.rule() 声明顺序）
      │
      ├─ 失败 → ToolResult(INVALID_ARGUMENTS, 详细错误信息)
      │
      └─ 通过 → execute(pojo)
```

校验发生在工具执行之前。校验失败直接返回 `ToolResult`，不进入 `execute()` 逻辑。

## Schema 生成

### 职责归属

Schema 生成器独立于校验器。两者各自扫描注解，职责不交叉：

| 组件 | 输入 | 输出 | 职责 |
|------|------|------|------|
| Schema 生成器 | `Class<?>`（带 @ToolParam） | 标准 JSON Schema 字符串 | 生成发送给 LLM 的 `tools[].function.parameters` |
| 校验器 | POJO 实例 + `Class<?>` | `ToolResult` 或通过 | 校验 LLM 返回的参数值 |

### 生成流程

1. 接收 `Class<?>`，检查是否标注 `@ToolParam`（否则拒绝）
2. 遍历所有字段，过滤带 `@ToolField` 的
3. 构建 JSON Schema 结构：
   - 顶层 `{"type": "object", "properties": {...}, "required": [...], "additionalProperties": false}`
   - 每个 `@ToolField` 转换为一个 property：
     - 字段名 → property key
     - 通过 `TypeConverterRegistry` 查找字段类型对应的 `TypeConverter`，调用 `convert(field, registry)` 得到类型 Schema 片段
     - `desc` → `description`
     - 将类型 Schema 片段与 description 合并为完整 property
4. `required` 数组生成：扫描 `rule` 中是否包含 `RequiredRule.class`，包含则加入 `required` 数组
5. 输出标准 JSON Schema 字符串

### 标准 JSON Schema 示例

```json
{
  "type": "object",
  "properties": {
    "path": {
      "type": "string",
      "description": "文件的路径，支持相对路径和绝对路径"
    },
    "lines": {
      "type": "array",
      "description": "读取文件的具体行，如 '112' 或 '112-119'",
      "items": {
        "type": "string"
      }
    }
  },
  "required": ["path"],
  "additionalProperties": false
}
```

### Rule → JSON Schema 映射

某些 Rule 会影响 JSON Schema 的生成（如 `RequiredRule` → 加入 `required` 数组）。这个映射逻辑由 Schema 生成器内部维护，Rule 接口本身不提供 `toJsonSchema()` 方法——保持 Rule 只关注校验的单一职责。

当新增一个需要影响 Schema 结构的 Rule 时，需同时修改 Schema 生成器。这个代价可接受。

### 提供商适配

Schema 生成器默认产出标准 JSON Schema（OpenAI 兼容）。其他 LLM 提供商的格式差异由 LLM Client 层负责转换，不侵入工具参数定义层。

## ToolDefinition 结构

`ToolDefinition` 不再携带 `JsonNode` 格式的参数描述，改为直接绑定参数类型和返回值类型：

```java
interface Tool {
    ToolDefinition definition();
    ToolResult execute(Object params);  // params 是校验后的 POJO
}

class ToolDefinition {
    String name;
    String description;
    Class<?> paramType;    // 参数 POJO 类（带 @ToolParam）
    Class<?> resultType;   // 返回值类型
}
```

## 关键设计决策

### 不做中间 Schema 对象模型

不定义通用的 JSON Schema 类型体系（`JsonSchemaType`、`SchemaProperty` 等 Java 类）来同时承载序列化和校验。理由：

- 序列化只需要标准 JSON Schema，直接生成字符串即可
- 校验逻辑中部分规则（如 `LinesRule` 的 `"119-112"` 非法）JSON Schema 根本无法表达
- 中间模型会引入额外的维护成本和转换逻辑

### required 的双重表达不冗余

`@ToolField` 上不设独立的 `required` 属性。`required` 信息通过 `rule = {RequiredRule.class}` 表达——Schema 生成器读到它时加入 `required` 数组，校验器读到它时执行非空校验。同一个注解，两个消费者各自解读。

### 反射代价可接受

Schema 生成器、校验器、Jackson 各自独立扫描注解，不做缓存。工具参数类字段数量少、调用频率低，反射开销不构成瓶颈。

### 第一版手写 TypeConverter，后续切换到 victools/jsonschema-generator

第一版自己实现 `TypeConverter` + `TypeConverterRegistry`，覆盖常用 Java 类型到 JSON Schema 的映射。目的是先把手写完整流程跑通，理解类型推导的细节。

后续版本替换为 [victools/jsonschema-generator](https://github.com/victools/jsonschema-generator)——该库已成熟处理所有 Java 类型 → JSON Schema 映射（含泛型嵌套、枚举、JSR-303 校验注解等），Spring AI 也用同一套方案。切换时将现有的 `TypeConverter` 替换为 victools 的 `Module` 机制（类比 `SpringAiSchemaModule`），注解扫描逻辑不动，仅替换类型映射的实现。

手写的 `TypeConverter` 注册中心本身就是面向接口编程的，替换为 victools 只需新增一个 `VictoolsTypeConverterRegistry` 实现即可。

## 设计演变

### 变更对比表

| 变更点 | 原始方向 | 最终方向 | 原因 |
|--------|---------|---------|------|
| 参数约束表达 | 自定义 Schema 格式（`desc`、`fields` 数组、`arrayItem`） | 注解驱动 POJO + Schema 生成器 | 自定义格式无法对接 OpenAI API；注解更贴近 Java 习惯 |
| Schema 格式 | 内部自有格式 | 默认标准 JSON Schema，LLM Client 做适配 | 降低内部复杂度，适配交给边界层 |
| 校验与 Schema 关联 | Rule 提供 `toJsonSchema()` | Schema 生成器独立维护映射 | 保持 Rule 单一职责，不耦合 Schema 生成逻辑 |
| ToolDefinition 参数 | 存 `JsonNode` 的 Schema 描述 | 存 `Class<?>` 参数类型 | 避免手拼 JSON，Schema 按需生成 |
| 参数校验 JSON Schema 不能表达的规则 | —— | 内置在 Rule 中，校验时执行 | 如 `"119-112"` 非法这类业务规则必须由代码校验 |
| 字段类型声明 | `@ToolField.type` + `FieldType` 嵌套注解体系 | Java 字段类型反射推导 + `TypeConverter` 注册中心 | `FieldType` 在 Java 类型系统外又建并行体系，维护成本高；`ArrayType.items` 的泛型信息在 class 字面量中丢失 |

### 原始方向

最初考虑定义一套面向人类文档的自定义 Schema 格式（用 `desc`、`fields` 数组、`arrayItem`、`examples` 等字段），同时用于文档生成和参数约束。该格式与 OpenAI 的函数调用参数格式不兼容。

### 问题分析

1. **Schema 格式兼容性**：自定义格式无法直接作为 OpenAI API `tools[].function.parameters` 使用，需额外转换层
2. **校验与 Schema 生成耦合**：最初考虑让 Rule 同时提供 Schema 片段，但会导致 Rule 职责膨胀，且多个 Rule 的 Schema 片段拼接复杂
3. **中间对象模型必要性**：最初考虑定义通用 JSON Schema Java 对象模型，但校验中的业务规则（如行范围合法性）JSON Schema 无法表达，中间模型收益有限

### 二次演变：移除 FieldType 类型体系

#### 原始方案

`@ToolField` 上通过 `type` 属性声明 `FieldType` 嵌套注解来指定字段类型。`FieldType` 内部定义了 `StringType`、`NumberType`、`BooleanType`、`ArrayType`、`ObjectType` 五个子注解来表达类型层级：

```java
@ToolField(
    name = "path",
    type = FieldType.StringType.class,
    desc = "文件的路径",
    rule = { RequiredRule.class }
)
private String path;

@ToolField(
    name = "lines",
    type = FieldType.ArrayType.class,
    desc = "读取文件的具体行",
    rule = { LinesRule.class }
)
private List<String> lines;
```

#### 问题分析

1. **维护两套并行类型体系**：`FieldType` 本身是一套类型枚举，与 Java 字段类型并⾏存在。字段声明了 `String path`，仍需额外标注 `type = FieldType.StringType.class`，冗余且容易不一致
2. **泛型信息丢失**：`type = FieldType.ArrayType.class` 无法携带 `items` 子类型信息（class 字面量丢失了注解成员值）。原设想语法 `@FieldType.Array(items = @FieldType.String)` 需要 `type` 属性接受注解实例而非 `Class<?>`，Java 语法层面难以干净实现
3. **扩展不便**：新增一个特殊格式（如 URL）需要同时在 `FieldType` 中新增子注解、在 Schema 生成器中增加分支，侵入点多

#### 最终方案

- 删除 `FieldType` 及其所有嵌套注解，`@ToolField` 不再携带 `type` 属性
- 通过 `Field.getGenericType()` 反射直接推导 JSON Schema 类型
- 引入 `TypeConverter` 接口 + `TypeConverterRegistry` 注册中心，默认覆盖常用类型
- `@ToolField.typeConverter` 作为自定义覆盖扩展点（类比 Jackson 的 `@JsonSerialize`）
