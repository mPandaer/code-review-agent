# 工具层接口设计

> 关联设计文档：[001-Agent-设计文档](./001-Agent设计文档.md)

## 设计原则

Tool 层保持厂商无关（vendor-neutral），不感知任何 LLM 供应商的协议格式。格式转换职责归属 LLM Client 层。

## 核心接口

### Tool

工具接口，抽象 Agent 可以使用的工具。

```java
interface Tool {
    ToolDefinition definition();
    ToolResult execute(JsonNode arguments);
}
```

- `definition()`：返回工具的元信息
- `execute(JsonNode)`：执行工具，接收 Jackson `JsonNode` 作为参数

### ToolDefinition

工具定义 POJO，持有厂商无关的结构化数据：

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | `String` | 工具名称，LLM 通过此名称发起 tool call |
| `description` | `String` | 工具功能描述，帮助 LLM 决定何时调用 |
| `inputSchema` | `JsonNode` | 入参 JSON Schema，描述参数结构和类型 |
| `outputDescription` | `JsonNode` | 返回值结构描述，JSON Schema 格式 |

> `ToolDefinition` 是 `Tool` 的一部分，通过 `Tool.definition()` 获取。

### ToolResult

工具执行结果：

| 字段 | 类型 | 说明 |
|------|------|------|
| `status` | `String` | 执行状态：`SUCCESS` / `INVALID_ARGUMENTS` / `EXECUTION_ERROR` |
| `message` | `String` | `SUCCESS` 时固定 `"SUCCESS"`，失败时为具体原因 |
| `data` | `Object` | 工具返回的结构化数据，各工具类型不同 |

### ToolRegistry

工具注册中心，Agent 可见的工具列表由 `ToolRegistry` 决定：

```java
interface ToolRegistry {
    List<ToolDefinition> definitions();
    Optional<Tool> find(String name);
}
```

- `definitions()`：返回所有已注册工具的定义列表
- `find(String)`：按名称查找工具实例

## 格式转换（归属 LLM Client 层）

由于不同 LLM 供应商的工具调用协议格式存在差异（如 OpenAI 使用专用 `tools` 参数，Anthropic 的 tool definition 格式略有不同），格式转换器放在 LLM Client 层：

1. **ToolDefinition 转换器**：`ToolDefinition` → 各 LLM 供应商的 tools 参数格式
2. **ToolResult 转换器**：`ToolResult` → LLM 协议中的 `ToolMessage`

Tool 层只产出 `ToolDefinition`（POJO）和 `ToolResult`，不关心最终呈现给 LLM 的格式。

## 设计演变

### 最终方案

| 变更点 | 旧设计 | 新设计 | 原因 |
|--------|--------|--------|------|
| `ToolDefinition` 格式 | `desc()` 返回 XML 字符串 | POJO，持有结构化字段 | 保持厂商无关，格式转换交给 LLM Client |
| `ToolRegistry` 方法 | `list()` 返回描述字符串列表 | `definitions()` 返回 `List<ToolDefinition>` | 返回结构化数据，LLM Client 层按需转换 |
| 格式转换位置 | Tool 层自己处理 | LLM Client 层 | Tool 层不应感知 LLM 供应商协议 |

### 原始方案（最初想法）

以下是设计讨论开始前的原始想法，保留于此以记录推导过程。

---

1. **Tool**：工具接口，用于抽象 Agent 可以使用的工具。有一个方法 `ToolResult execute(args)`
2. **ToolDefinition**：工具定义，有四部分内容：工具名字、工具描述、入参描述、返回值描述。提供一个方法 `desc()` 返回 XML 格式字符串，包含工具的描述、入参和返回的结构定义。
3. **ToolResult**：工具执行结果
    - `String status`：`SUCCESS` / `INVALID_ARGUMENTS` / `EXECUTION_ERROR`
    - `String message`：成功时固定 `"SUCCESS"`，失败时为具体原因
    - `Object data`：各工具返回结果不同，使用 `Object`
4. **ToolRegistry**：工具注册中心
    - `List<String> list()`：返回工具描述信息（含入参、返回、描述），主要放在 Agent 的 System prompt 中
    - `Optional<Tool> find(String name)`：按名称查找工具实例

> 原始方案中 `ToolDefinition` 是 `Tool` 的一部分。

#### 原始方案的问题

经过苏格拉底式讨论，识别出以下问题并逐一修正：

1. **`desc()` 返回 XML** → XML 不能直接用于 OpenAI 的 `tools` 参数（JSON Schema 格式），如果未来扩展其他 LLM 厂商格式还可能不同。改为 POJO 持有结构化数据，格式转换交给 LLM Client 层。
2. **`list()` 返回描述字符串** → 返回字符串不如返回结构化数据灵活，LLM Client 不能按需转换。改为 `definitions()` 返回 `List<ToolDefinition>`。
3. **格式转换放在 Tool 层** → 会让 Tool 层感知 LLM 供应商协议，违反分层职责。改为 LLM Client 层负责。
4. **描述放 System prompt** → LLM 实际通过 API 的 `tools` 参数获取工具定义，不通过 System prompt。改为转换器产出 LLM 原生格式。
