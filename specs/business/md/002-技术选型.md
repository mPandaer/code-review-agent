# 技术选型

> 关联设计文档：[001-Agent-设计文档](./001-Agent设计文档.md)

## 选型总览

| 领域 | 选型 | 理由 |
|------|------|------|
| 构建工具 | Gradle (Kotlin DSL) | 用户指定 |
| Java 版本 | Java 8+ | 用户指定，保证广泛兼容性 |
| CLI 参数解析 | picocli | Java 生态最现代的 CLI 库，注解+编程式双 API，支持子命令、自动补全、ANSI 彩色帮助信息 |
| 依赖注入 | Manual DI（无框架） | 对象图很小（Agent → LlmClient + ToolRegistry + Session），手动 DI 足够，避免框架学习成本和启动开销 |
| HTTP 客户端 | OkHttp | 原生支持 streaming response body，SSE 协议处理成熟，LLM 流式响应场景的最佳选择 |
| JSON 处理 | Jackson | Java 生态功能最全的 JSON 库，同时支持 YAML（通过 dataformat-yaml 模块），`Tool.execute(JsonNode)` 天然匹配 |
| YAML 配置 | Jackson YAML 模块 | 复用 Jackson 生态，不需要单独引入 SnakeYAML |
| 日志 | SLF4J + Logback | SLF4J 提供门面（代码零耦合），Logback 配置简单、性能足够，是 Spring Boot 默认方案 |
| 测试 | JUnit 5 | Java 生态标准测试框架 |

## 各选型详细说明

### CLI 参数解析：picocli

**不选其他方案的原因：**

- **Commons CLI**：传统编程式 API，功能较弱，不支持子命令、自动补全。只适合极简单的命令行工具。
- **JCommander**：纯注解式，功能和活跃度不如 picocli（最后发布 2022年1月）。
- **argparse4j**：基于 Python argparse 的移植，类型安全性差，最后发布 2021年4月。

**选择 picocli 的理由：**

- 注解式 + 编程式双 API，灵活性最高
- 子命令支持（如 `pr session list`）
- ANSI 彩色帮助信息
- 自动补全脚本生成
- 活跃维护（最后发布 2024年5月）

### 依赖注入：Manual DI

对象依赖关系简单：

```
Agent
├── LlmClient      (稳定，编译期确定)
├── ToolRegistry   (稳定，运行时注册工具)
│   └── Tool[]
└── Session         (运行时创建)
```

对象数量少、依赖层次浅，手动 DI 完全可控。如果未来扩展需要，切换到 Guice 只需修改组装层，不影响核心代码。

### HTTP 客户端：OkHttp

**为什么不是 Retrofit？**

Retrofit 解决的是"把 REST API 多个端点映射成类型安全的 Java 接口"，你的 `LlmClient` 只有一个方法（`chat`），一个端点。为这一个调用引入完整的 Retrofit 依赖链不值得。

更关键的是 **Streaming**。OpenAI 的 streaming 使用 SSE 协议——逐行返回 `data: {...}` 流。Retrofit 需要自定义 `Converter.Factory` 和 `CallAdapter.Factory` 才能处理 SSE，适配代码量比直接用 OkHttp 手写 streaming 逻辑更多。

**为什么不是 Apache HttpClient 5 / Java 11 HttpClient？**

| 特性 | OkHttp | Apache HttpClient 5 | Java 11 HttpClient |
|------|--------|---------------------|-------------------|
| Streaming body | ✅ 原生支持 | ⚠️ 需要额外处理 | ✅ `BodySubscribers.ofLines()` |
| SSE 处理 | ✅ 成熟 | ❌ 需手写大量代码 | ⚠️ 需手写 `data:` 前缀解析 |
| LLM 生态采用 | ✅ 多个 Java LLM SDK 底层 | ❌ 很少 | ❌ 很少 |

### JSON 处理：Jackson

**为什么不是 Gson？**

- Gson 简单，但功能不够：缺少 `JsonNode` 的树模型（你的 `Tool.execute(JsonNode)` 已经在用）
- Jackson 同时支持 YAML，一个依赖解决两个问题
- Jackson 是 Spring Boot 默认选择，生态最广、文档最多

### 日志：SLF4J + Logback

**为什么不是 Log4j2？**

Log4j2 的杀手锏是异步日志和无垃圾模式（garbage-free logging），这是为高并发服务端设计的。Code Review Agent 是 CLI 工具，日志量极小，这些优化用不上。

Logback 和 SLF4J 是同一作者（Ceki Gülcü），天然无缝集成，配置更简单。如果未来真的出现性能瓶颈，通过 SLF4J 门面切换到 Log4j2 只需改依赖和配置文件，业务代码零改动。

## 技术选型与架构的映射

```
picocli ──────────► CLI 层
Manual DI ────────► Agent Core 层（组装 Agent + ToolRegistry + Session）
Jackson ──────────► 全项目（JSON 请求解析 + JSON 响应解析 + YAML 配置读取）
OkHttp ───────────► LLM Client 层（HTTP 传输 + SSE Streaming）
SLF4J + Logback ──► 全项目
JUnit 5 ──────────► 全项目测试
```
