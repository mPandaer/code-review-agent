Code Review Agent 4 份文档

设计文档

Code Review Agent 的架构设计与技术选型，基于 Java 8+ 构建，三层架构解耦 CLI / Agent Core / LLM Client。

SECTION 01

Agent 整体设计

第一版范围：不依赖 Agent 框架，Java 实现，核心机制为 Function Call / Tool Use。

核心结论

三层架构成立，Agent Core 内部必须继续拆分

外部分 CLI、Agent Core、LLM API Client 三层。最容易崩掉的地方在 Agent Core 内部 —— 如果 Session、RunState、RunConfig、ToolRegistry 的生命周期混在一起，会导致上下文污染、配置语义不清、工具调用逻辑散落。

01 层

CLI

  • ▸ 解析用户命令和交互输入
  • ▸ 创建、选择或加载 Session
  • ▸ 调用 Agent.run(session, userInput)
  • ▸ 渲染 AgentRunResult
02 层

Agent Core

  • ▸ 维护 Agent Loop
  • ▸ 追加 user / assistant / tool message
  • ▸ 根据 LLM 返回决定是否调用工具
  • ▸ 处理最大轮次和运行结果
03 层

LLM API Client

  • ▸ 请求转换为供应商格式
  • ▸ 发送 LLM API 请求
  • ▸ 响应转换为 LlmResponse
  • ▸ 不关心 Agent Loop
接口草图

关键接口

LlmClient.chat(request)
Tool.definition()
Tool.execute(arguments)
ToolRegistry.definitions()

推荐结构

CLI CommandParser ConsoleRenderer Agent Core Agent Session SessionConfig RunConfig RunState AgentRunResult LLM LlmClient LlmRequest LlmResponse AssistantMessage ToolCall Tool Tool ToolDefinition ToolRegistry ToolResult ToolMessageFactory

Agent Loop 流程

  1. 追加用户消息。 agent.run(session, userInput) 开始时先追加为 UserMessage，确保 LLM 第一轮能看到新任务。
  2. 冻结运行配置。 从 SessionConfig 复制生成 RunConfig，创建空的 RunState。
  3. 调用 LLM。 使用当前消息列表、工具定义和配置构造 LlmRequest。
  4. 保存 assistant 消息。 把 LlmResponse.assistantMessage 追加进 Session.messages。
  5. 处理工具调用。 存在 tool calls 时按顺序执行，结果转换成 ToolMessage 追加进消息列表。
  6. 判断结束。 无 tool calls → 返回结果；达到 maxRounds → 返回 MAX_ROUNDS_EXCEEDED。不追加控制消息。
长期上下文

Session

Session 保存会话级历史，message list 是 LLM 可见上下文主载体。ToolResult → ToolMessage → 消息列表。maxRounds exceeded 不写入消息列表。

单次运行

RunState

RunState 保存当前轮次、是否结束等临时状态。currentRound 只属于一次 run。RunConfig 从 SessionConfig 快照生成，运行中不受 Session 修改影响。

运行结果

AgentRunResult

status → COMPLETED / MAX_ROUNDS_EXCEEDED / FAILED
finalMessage → 成功时的最终回答
roundsUsed → 消耗轮次数

关键决策

工具调用

ToolRegistry 只负责注册和查找。Tool 自己负责 JSON 反序列化和校验。参数错误包装为 ToolResult 回传 LLM。未知工具由 Agent Core 转为失败 ToolResult。

⚠️ 三个必须守住的边界

  • 不要把 Session 和 RunState 混在一起 —— 会导致当前轮次、最大轮次错误、临时状态污染长期上下文。
  • 不要把 ToolResult 和 ToolMessage 混在一起 —— 会让工具领域模型和 LLM 协议格式耦合。
  • 不要让 Agent Core 解析供应商原始响应 —— 否则多 LLM Client 实现时，Agent Core 会变成响应格式适配层。
SECTION 02

技术选型

关联设计文档：001-Agent 设计文档

领域 选型 理由
构建工具 Gradle (Kotlin DSL) 用户指定
Java 版本 Java 8+ 用户指定，保证广泛兼容性
CLI 参数解析 picocli 注解+编程式双 API，子命令、自动补全、ANSI 彩色帮助信息
依赖注入 Manual DI 对象图小，手动 DI 足够，避免框架学习成本和启动开销
HTTP 客户端 OkHttp 原生支持 streaming body，SSE 协议处理成熟
JSON 处理 Jackson 功能最全，同时支持 YAML，Spring Boot 默认选择
日志 SLF4J + Logback SLF4J 提供门面，Logback 配置简单，Spring Boot 默认方案
测试 JUnit 5 Java 生态标准测试框架

选型对比详情

CLI 解析

picocli vs 其他

不选：Commons CLI 功能弱、JCommander 已停更（2022）、argparse4j 类型安全性差。

选 picocli：注解+编程式双 API，子命令支持，ANSI 彩色帮助，自动补全生成，活跃维护（2024.5）。

依赖注入

Manual DI

对象图极小：Agent → LlmClient + ToolRegistry + Session，依赖层次浅。如果未来需要，切换到 Guice 只需改组装层，核心代码不动。

HTTP vs 其他

OkHttp vs Retrofit

不选 Retrofit：LlmClient 只有一个 chat 方法，不值得引入完整依赖链。SSE streaming 需要自定义 Converter 和 CallAdapter，适配代码量比直接用 OkHttp 更多。

HTTP 对比

OkHttp vs Apache/Java 11

OkHttp 胜出：Streaming body 原生支持、SSE 处理成熟、LLM 生态广泛采用。
Apache HttpClient 5 和 Java 11 HttpClient 在 SSE 处理上需手写大量代码。

JSON

Jackson vs Gson

不选 Gson：缺少 JsonNode 树模型（Tool.execute(JsonNode) 依赖）。

选 Jackson：同时支持 YAML，一个依赖解决两个问题，Spring Boot 默认，生态最广。

日志

Logback vs Log4j2

不选 Log4j2：异步日志和无垃圾模式为高并发服务端设计，CLI 工具用不上。

选 Logback：与 SLF4J 同作者，无缝集成。通过 SLF4J 门面可随时切换。

技术选型与架构的映射

picocli → CLI 层
Manual DI → Agent Core 层（组装 Agent + ToolRegistry + Session）
Jackson → 全项目（JSON 解析 + YAML 配置）
OkHttp → LLM Client 层（HTTP 传输 + SSE Streaming）
SLF4J + Logback → 全项目
JUnit 5 → 全项目测试
SECTION 03

工具层接口设计

关联设计文档：001-Agent 设计文档

设计原则

厂商无关，格式转换归 LLM Client 层

Tool 层保持 vendor-neutral，不感知任何 LLM 供应商协议格式。ToolDefinition 为纯 POJO，格式转换器归属 LLM Client 层。

接口

Tool

ToolDefinition definition()
ToolResult execute(JsonNode)

POJO

ToolDefinition

name — 工具名称
description — 功能描述
inputSchema — 入参 JSON Schema
outputDescription — 返回值结构（JsonNode）

结果

ToolResult

status — SUCCESS / INVALID_ARGUMENTS / EXECUTION_ERROR
message — 成功 / 失败原因
data — 结构化返回数据

注册中心

ToolRegistry

List<ToolDefinition> definitions()
Optional<Tool> find(String name)

格式转换（归属 LLM Client 层）

转换器 1

ToolDefinition → LLM 格式

将 vendor-neutral 的 ToolDefinition 转换为各 LLM 供应商的 tools 参数格式。OpenAI 使用 JSON Schema 格式的 tools 参数，Anthropic 格式略有不同。

转换器 2

ToolResult → ToolMessage

将 ToolResult 转换为 LLM 协议中的 ToolMessage。OpenAI 用 role: "tool"，Anthropic Claude 用 role: "user" + tool_result 块。

与之前设计的差异

变更点 旧设计 新设计 原因
ToolDefinition 格式 desc() 返回 XML POJO，结构化字段 保持厂商无关，转换交 LLM Client
ToolRegistry 方法 list() 返回描述字符串 definitions() 返回 List 返回结构化数据，LLM Client 按需转换
格式转换位置 Tool 层自己处理 LLM Client 层 Tool 层不应感知 LLM 供应商协议

原始方案

以下是设计讨论前的原始想法，保留于此记录推导过程。

原始设计

desc() → XML

ToolDefinition 用 desc() 方法返回 XML 格式字符串，包含工具描述、入参和返回值结构。

原始设计

list() → System prompt

ToolRegistry 用 list() 返回描述字符串列表，直接放入 Agent 的 System prompt 中。

原始设计

Tool 层自转换

格式转换逻辑放在 Tool 层内部，Tool 自身负责产出各 LLM 厂商所需格式。

讨论中发现的问题

  • XML 不能直接用于 OpenAI tools 参数 — LLM 使用 JSON Schema 格式，未来其他厂商格式也可能不同。改为 POJO，转换交 LLM Client。
  • list() 返回字符串不灵活 — 改为 definitions() 返回结构化数据，LLM Client 按需转换。
  • Tool 层感知 LLM 协议违反分层 — 格式转换移到 LLM Client 层。
  • 描述放 System prompt 不正确 — LLM 实际通过 API 的 tools 参数获取工具定义。
SECTION 04

工具参数注解

关联设计文档：001-Agent 设计文档 · 004-工具参数注解

核心结论

注解驱动 POJO，不做中间通用 Schema 对象模型

工具参数约束通过 @ToolParam 和 @ToolField 注解声明在 POJO 上。Schema 生成器、校验器、Jackson 反序列化各自独立扫描注解，不做缓存、不共享中间对象。默认产出标准 JSON Schema，不同 LLM 提供商的格式适配由 LLM Client 层负责。

类级注解

@ToolParam

标注参数类，声明这是工具的参数对象。

name — 工具参数的中文描述

字段级注解

@ToolField

name — 字段名
desc — 字段描述
rule — 校验规则数组（按序执行）
typeConverter — 自定义类型转换器（可选）

类型推导

TypeConverter 注册中心

不再维护 FieldType 类型体系。通过反射从 Java 字段类型推导 JSON Schema 类型。
TypeConverter — 类型转换器接口
TypeConverterRegistry — 注册中心，默认支持 String / 数字 / List / Set / 枚举

校验规则

Rule 接口

Boolean validate(Object value)

Array 类型时 value 为 List。按 @ToolField.rule() 声明顺序执行，任一失败立即返回。

示例：ReadFileParams

@ToolParam(name = "读文件工具的参数") public class ReadFileParams { @ToolField( name = "path", desc = "文件的路径，支持相对路径和绝对路径", rule = { RequiredRule.class } ) private String path; @ToolField( name = "lines", desc = "读取文件的具体行，如 '112' 或 '112-119'", rule = { LinesRule.class } ) private List<String> lines; }
LinesRule

合法值

"112" → 读单行
"112-119" → 读行范围（起始 ≤ 结束）

LinesRule

非法值

"119-112" → 起始 > 结束 ❌
"112~119" → 分隔符非 "-" ❌

工具调用链路

  1. LLM 返回 JSON。 Agent Loop 收到 LLM 的 Function Call 响应，包含 JSON 参数。
  2. Jackson 反序列化。 根据 ToolDefinition.paramType 将 JSON 反序列化为 POJO 实例。
  3. Rule 链校验。 遍历 @ToolField.rule() 数组，按声明顺序执行。任一失败 → 返回 ToolResult(INVALID_ARGUMENTS)。
  4. 执行工具。 校验通过后调用 execute(pojo)，返回 ToolResult。

Schema 生成

职责分离

Schema 生成器 vs 校验器

Schema 生成器只管产出标准 JSON Schema，校验器只管跑 Rule。两者独立扫描注解，职责不交叉。Rule 接口不提供 toJsonSchema() 方法，保持单一职责。

生成流程

Class → JSON Schema

1. 检查是否标注 @ToolParam
2. 遍历字段，过滤 @ToolField
3. 通过 TypeConverterRegistry 查找类型转换器
4. 扫描 rule 中是否有 RequiredRule → required 数组

映射策略

Rule → Schema

映射由 Schema 生成器内部维护（如 RequiredRule → required 数组）。新增 Rule 时若需影响 Schema 结构，需同时修改生成器——这个代价可接受。

提供商适配

交 LLM Client 层

Schema 生成器默认产出标准 JSON Schema（OpenAI 兼容）。其他提供商格式差异由 LLM Client 层转换，不侵入工具参数定义层。

生成的 JSON Schema 示例

{ "type": "object", "properties": { "path": { "type": "string", "description": "文件的路径，支持相对路径和绝对路径" }, "lines": { "type": "array", "description": "读取文件的具体行，如 '112' 或 '112-119'", "items": { "type": "string" } } }, "required": ["path"], "additionalProperties": false }

ToolDefinition 结构变更

变更

不再携带 JsonNode Schema，改用 Class<?> 绑定

ToolDefinition 不再存储 JsonNode 格式的参数描述。改为直接绑定 paramType（参数 POJO 类的 Class<?>）和 resultType。Schema 按需从注解生成，避免了手拼 JSON 的维护负担。

设计演变

变更点 原始方向 最终方向 原因
参数约束表达 自定义 Schema 格式（desc、fields 数组、arrayItem） 注解驱动 POJO + Schema 生成器 自定义格式无法对接 OpenAI API；注解更贴近 Java 习惯
Schema 格式 内部自有格式 默认标准 JSON Schema，LLM Client 做适配 降低内部复杂度，适配交给边界层
校验与 Schema 关联 Rule 提供 toJsonSchema() Schema 生成器独立维护映射 保持 Rule 单一职责，不耦合 Schema 生成逻辑
ToolDefinition 参数 存 JsonNode 的 Schema 描述 存 Class<?> 参数类型 避免手拼 JSON，Schema 按需生成
JSON Schema 之外的校验 — 内置于 Rule 中，校验时执行 如 "119-112" 非法这类业务规则必须由代码校验
字段类型声明 @ToolField.type + FieldType 嵌套注解体系 Java 字段类型反射推导 + TypeConverter 注册中心 FieldType 在 Java 类型系统外又建并行体系，维护成本高；ArrayType.items 泛型信息在 class 字面量中丢失

原始方向

最初考虑定义一套面向人类文档的自定义 Schema 格式（用 desc、fields 数组、arrayItem、examples 等字段），同时用于文档生成和参数约束。该格式与 OpenAI 的函数调用参数格式不兼容。

讨论中发现的问题

  • 自定义格式不兼容 OpenAI API — 需额外转换层，不如直接从注解生成标准 JSON Schema。
  • 校验≠JSON Schema 表达范围 — LinesRule 的 "119-112" 非法判断 JSON Schema 无法表达，必须由代码校验。中间模型收益有限。
  • required 双重表达可合并 — RequiredRule 同时服务于 Schema 生成（加入 required 数组）和校验（非空检查），不设独立的 required 属性。
  • Rule 不应兼任 Schema 片段 — 多个 Rule 的片段拼接复杂，且违反单一职责。改为 Schema 生成器内部维护映射。
  • ToolDefinition 存 JsonNode 不够简洁 — 改为存 Class<?>，Schema 按需从注解生成。
  • 第一版手写 TypeConverter，后续切 victools — 先跑通流程理解细节，接口设计已预留替换空间（面向 TypeConverter 接口编程）。后续替换为 Spring AI 同款的 victools/jsonschema-generator。

二次演变：移除 FieldType 类型体系

原始方案：FieldType 嵌套注解

@ToolField 通过 type 属性声明 FieldType 嵌套注解来指定字段类型。 FieldType 内部定义了 StringType、NumberType、BooleanType、ArrayType、ObjectType 五个子注解。

@Retention(RUNTIME) @Target({}) public @interface FieldType { @interface StringType {} @interface NumberType {} @interface BooleanType {} @interface ArrayType { Class<? extends Annotation> items(); } @interface ObjectType {} }

讨论中发现的问题

  • 维护两套并行类型体系 — 字段声明了 String path，仍需额外标注 type = FieldType.StringType.class，冗余且易不一致。
  • 泛型信息丢失 — type = FieldType.ArrayType.class 无法携带 items 子类型信息。class 字面量丢失注解成员值，Java 语法难以干净实现。
  • 扩展不便 — 新增特殊格式（如 URL）需同时修改 FieldType 和 Schema 生成器，侵入点多。
  • 解决方案 — 删除 FieldType，通过 Field.getGenericType() 反射推导 + TypeConverter 注册中心替代。@ToolField.typeConverter 留作自定义扩展点。
