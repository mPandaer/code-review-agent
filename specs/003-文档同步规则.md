# 设计文档规则

本文档包含两类规则：设计文档的写作规范，以及 HTML 索引的同步规范。

## 适用场景

当用户在 `specs/business/md/` 目录下新增或更新确认的设计文档时，触发此规则。

---

## 一、设计文档写作规范

### 保留原始设计

当一份设计文档经过讨论后发生变更，必须在文档末尾保留原始方案——而非直接覆盖。

格式要求：在文档末尾增加「设计演变」章节，包含：

1. **变更对比表**：列出新旧设计的差异点及原因
2. **原始方案**：讨论前的完整原始设计
3. **问题分析**：每个变更点对应的讨论结论

目的：让读者（包括未来的自己）能看到设计是怎么一步一步推导出来的，而非只看到一个最终结果。

### 文档格式

- 文档编号三位数字前缀（`001-`、`002-` 等）
- 顶部注明关联设计文档的链接
- 关键结论用表格或高亮块突出

---

## 二、HTML 索引同步规范

### 要求

当一份确认的文档被写入 `specs/business/md/` 目录（新增或更新），必须同步更新 `specs/business/index.html`。

该 HTML 文档的要求：

1. **整合展示**：汇总 `specs/business/md/` 目录下所有 `.md` 文档的核心内容，而非简单罗列文件名。
2. **视觉风格**：严格遵循 `specs/business/DESIGNS.md` 中定义的设计令牌、布局组件和配色规则。
3. **保持一致**：HTML 内容必须与对应的 `.md` 文档保持同步，用户更新任何 `.md` 文档后都需要同步更新 `index.html`。

> 视觉设计规范详见 `@specs/business/DESIGNS.md`——包含 Atomic CSS 变量、布局组件、配色轮换规则和响应式断点。

### 触发方式

建议通过 Git hook（如 `post-commit` 或自定义脚本）在检测到 `specs/business/md/` 目录变更时自动提醒同步，而非依赖人工记忆。
